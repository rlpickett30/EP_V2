Status: Draft from drawings, not code-verified

Document rule:

Purpose:

Subscribing subsystem(s):

============================================================
GUI / MODE COMMAND EVENTS
=========================

DETECTION_MODE_CHANGE:
Purpose:
GUI command requesting a change to the active detection method.

Publishing subsystem:
GUI Interface / GUI Command Manager
Server Communication-listener republishes it after receiving it from GUI.

Subscribing subsystem(s):
Server Communication-listener
Platform Registry / Mode Manager
Event Journal

Reason for subscription:
Communication-listener must recognize the inbound GUI command and place it onto the server event bus.
Platform Registry validates and translates the command into a TDOA-facing mode command.
Event Journal records the inbound command.

Path / timeline:
GUI command button
-> GUI Communication
-> UDP to Server
-> Server Communication-listener
-> Platform Registry / Mode Manager
-> TDOA_CHANGE_MODE
-> TDOA

Current status:
Partially broken.
The server receives the event over Communication, but Communication currently logs it as an unknown inbound event.

Notes / legacy aliases:
Legacy alias: GUI_DETECTION_MODE_CHANGE
Canonical name recommendation: DETECTION_MODE_CHANGE
Likely payload values: ENABLE_PATTERN_ONSET, ENABLE_ENERGY_ONSET
Do not redesign this into a combined mode event for V2.0.

Server Communication receives them, but logs them as unknown inbound events.

Expected handling:

Platform Registry / Mode Manager

NETWORK_MODE_CHANGE

DETECTION_MODE_CHANGE
