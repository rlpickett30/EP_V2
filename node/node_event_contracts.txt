============================================================
SERVER INBOUND / NODE COMMAND EVENTS
====================================

TDOA_REQUEST:
Purpose:
Server command requesting that the node perform a synchronized TDOA recording.

Publishing subsystem:
Node Communication-listener, after receiving it from the server.
Server TDOA is the original platform source.

Subscribing subsystem(s):
Node Communication-listener
Node Microphone
Node Event Journal

Reason for subscription:
Communication-listener receives and republishes the server command onto the node event bus.
Microphone must start the recording workflow.
Event Journal records the inbound command.

Path / timeline:
Server TDOA
-> Server Communication-Sender
-> Node Communication-listener
-> TDOA_REQUEST
-> Node Microphone
-> TDOA_RECORDING

Current status:
Drawn as Communication-listener publication and Microphone subscription.

Notes / legacy aliases:
None identified.
This is a command event coming from server to node.

SEND_NODE_CHANGE_MODE:
Purpose:
Server command instructing the node to change an operating mode.

Publishing subsystem:
Node Communication-listener, after receiving it from the server.
Server Platform Registry / Mode Manager is the original platform source.

Subscribing subsystem(s):
Node Communication-listener
Node Communication internal mode handler
Node Event Journal

Reason for subscription:
Communication-listener receives and republishes the server mode-change command.
Node Communication or mode logic must apply the requested node-side change.
Event Journal records the inbound command.

Path / timeline:
GUI NETWORK_MODE_CHANGE
-> Server Platform Registry
-> SEND_NODE_CHANGE_MODE
-> Server Communication-Sender
-> Node Communication-listener
-> node mode handling

Current status:
Drawn as Communication-listener server-inbound publication and Journal subscription.

Notes / legacy aliases:
Name is server-oriented, but it is currently drawn as the node-received command.
Possible future cleanup candidate: NODE_CHANGE_MODE or NODE_MODE_CHANGE.
Do not rename until code scan confirms usage.

============================================================
MICROPHONE EVENTS
=================

PPS_STATE:
Purpose:
Node state event reporting PPS lock or PPS availability.

Publishing subsystem:
Node RTK subsystem

Subscribing subsystem(s):
Node Microphone
Node Communication-Sender
Node Event Journal

Reason for subscription:
Microphone needs PPS state to know whether synchronized recording can use PPS timing.
Communication-Sender sends PPS state to server.
Event Journal records the state update.

Path / timeline:
RTK/PPS driver
-> RTK subsystem
-> PPS_STATE
-> Microphone timing logic
-> Communication-Sender
-> server

Current status:
Drawn as RTK publication, Microphone subscription, Communication-Sender subscription, and Journal subscription.

Notes / legacy aliases:
None identified.

TDOA_RECORDING:
Purpose:
Node recording response produced after a TDOA request.

Publishing subsystem:
Node Microphone

Subscribing subsystem(s):
Node Communication-Sender
Node Event Journal

Reason for subscription:
Communication-Sender sends the recording response to the server.
Event Journal records the recording event.

Path / timeline:
TDOA_REQUEST
-> Node Microphone
-> TDOA_RECORDING
-> Communication-Sender
-> server

Current status:
Drawn as Microphone publication and Communication-Sender subscription.

Notes / legacy aliases:
Need to confirm payload type: audio data, file path, metadata, timestamp bundle, or staged recording reference.

RECORDING_AVAILABLE:
Purpose:
Node-local notice that a recording is available for BirdNET analysis.

Publishing subsystem:
Node Microphone

Subscribing subsystem(s):
Node BirdNet
Node Event Journal

Reason for subscription:
BirdNet needs local audio availability before running detection.
Event Journal records local recording availability.

Path / timeline:
Microphone capture completes
-> RECORDING_AVAILABLE
-> BirdNet
-> AVIS_LITE

Current status:
Drawn as Microphone publication and BirdNet subscription.

Notes / legacy aliases:
This appears node-local.
Do not confuse this with TDOA_RECORDING, which is the server-facing recording response.

============================================================
BIRDNET EVENTS
==============

AVIS_LITE:
Purpose:
Lightweight bird detection event.

Publishing subsystem:
Node BirdNet

Subscribing subsystem(s):
Node Communication-Sender
Node Event Journal
Server Communication-listener, after transmission

Reason for subscription:
Communication-Sender sends the detection to the server.
Event Journal records the detection.
Server receives it for Registry verification and possible TDOA triggering.

Path / timeline:
RECORDING_AVAILABLE
-> Node BirdNet
-> AVIS_LITE
-> Node Communication-Sender
-> server

Current status:
Drawn as BirdNet publication and Communication-Sender subscription.

Notes / legacy aliases:
None identified.
Server promotes this to SERVER_AVIS_LITE for GUI/database paths.
TDOA may also use raw AVIS_LITE as a candidate trigger after server receipt.

NODE REGISTRATION EVENTS
========================

NODE_REGISTER:
Purpose:
Node registration event sent to the server.

Publishing subsystem:
Node Main

Subscribing subsystem(s):
Node Communication-Sender
Node Event Journal
Server Communication-listener, after transmission

Reason for subscription:
Communication-Sender sends node registration to the server.
Event Journal records the local registration.
Server Registry records or refreshes the node.

Path / timeline:
Node Main startup
-> NODE_REGISTER
-> Node Communication-Sender
-> server
-> Platform Registry
-> SERVER_NODE_REGISTER

Current status:
Drawn as Node Main publication and Communication-Sender subscription.

Notes / legacy aliases:
None identified.
Node registration ownership should remain with node_register.py / Node Main path, not RTK or sensor subsystems.

COMMUNICATION EVENTS
====================

NETWORK_CONNECTED:
Purpose:
Node Communication status event indicating that a network path is connected.

Publishing subsystem:
Node Communication-listener or node Communication state logic

Subscribing subsystem(s):
Node Communication internal logic
Node Event Journal

Reason for subscription:
Communication internals update local connection state.
Event Journal records node-side network availability.

Path / timeline:
Node Communication detects connection available
-> NETWORK_CONNECTED
-> Event Journal / Communication internals

Current status:
Drawn as Communication internal event and Journal subscription.

Notes / legacy aliases:
Need code verification for whether this means Wi-Fi, LoRa, server UDP, or any active transport.

NETWORK_DISCONNECTED:
Purpose:
Node Communication status event indicating that a network path is disconnected.

Publishing subsystem:
Node Communication-listener or node Communication state logic

Subscribing subsystem(s):
Node Communication internal logic
Node Event Journal

Reason for subscription:
Communication internals update local connection state.
Event Journal records node-side network loss.

Path / timeline:
Node Communication detects connection loss
-> NETWORK_DISCONNECTED
-> Event Journal / Communication internals

Current status:
Drawn as Communication internal event and Journal subscription.

Notes / legacy aliases:
Need code verification for whether this means Wi-Fi, LoRa, server UDP, or any active transport.

EVENT_SENT:
Purpose:
Node Communication-Sender confirmation that an outbound message was sent or handed to transport.

Publishing subsystem:
Node Communication-Sender

Subscribing subsystem(s):
Node Event Journal

Reason for subscription:
Event Journal records outbound node traffic.

Path / timeline:
Node Communication-Sender receives outbound event
-> sends to server
-> EVENT_SENT
-> Node Event Journal

Current status:
Drawn as Communication-Sender publication and Journal subscription.

Notes / legacy aliases:
Need to confirm whether EVENT_SENT means attempted send or successful transport handoff.
