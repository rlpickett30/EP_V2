============================================================
GUI / MODE COMMAND EVENTS
=========================

NETWORK_MODE_CHANGE:
Purpose:
GUI command requesting a network/transport mode change, such as Wi-Fi or LoRa.

Publishing subsystem:
GUI Interface

Subscribing subsystem(s):
GUI Communication-Sender
GUI Event Journal
Server Communication-listener, after UDP transmission

Reason for subscription:
Communication-Sender must transmit the command to the server.
Event Journal records that the GUI published the command.
Server Communication-listener receives the command as GUI inbound traffic.

Path / timeline:
GUI Interface button/action
-> NETWORK_MODE_CHANGE
-> GUI Communication-Sender
-> UDP to Server
-> Server Communication-listener
-> Server Platform Registry / Mode Manager

Current status:
Drawing is correct.
Code likely has a naming or subscription mismatch in GUI Interface or GUI Communication-Sender.

Notes / legacy aliases:
Legacy alias: GUI_NETWORK_MODE_CHANGE
Canonical name: NETWORK_MODE_CHANGE
Do not use the GUI_ prefix as canonical.

IMMEDIATE GUI BUG TARGET
========================

Problem:
The GUI drawing uses canonical mode names:

NETWORK_MODE_CHANGE
DETECTION_MODE_CHANGE
FEATURE_MODE_CHANGE

But the server sees mode-change traffic as unknown inbound events.

Current diagnosis:
The likely mismatch is in GUI Interface code or GUI Communication-Sender subscriptions.

Expected GUI behavior:
Interface publishes:
NETWORK_MODE_CHANGE
DETECTION_MODE_CHANGE
FEATURE_MODE_CHANGE

Communication-Sender subscribes to:
NETWORK_MODE_CHANGE
DETECTION_MODE_CHANGE
FEATURE_MODE_CHANGE

Communication-Sender sends those exact event_type values to server.

Legacy aliases to scan for:
GUI_NETWORK_MODE_CHANGE
GUI_DETECTION_MODE_CHANGE
GUI_FEATURE_MODE_CHANGE

Canonical names:
NETWORK_MODE_CHANGE
DETECTION_MODE_CHANGE
FEATURE_MODE_CHANGE
